## Where the project is now

**Stack:** Vite + React + TypeScript (strict), TanStack Router + Query, `@supabase/supabase-js`, Zod, MSW, Vitest + RTL + Fishery + faker, Biome (lint+format), pgTAP for SQL tests. Node 20+. Package manager: npm (lockfile committed).

**The six CI gates that exist locally and should run in GH Actions:**

| Command | What it checks | Approx. local runtime |
|---|---|---|
| `npm run lint` | Biome lint + format check across `src/` and `scripts/` | <1s |
| `npx tsc -b --noEmit` | TypeScript typecheck | ~3s |
| `npm test` | Vitest suite (120 tests across 26 files, jsdom + MSW) | ~5s |
| `npm run check:schema` | Regenerates the JSON Schema from Zod and exits non-zero on drift | <1s |
| `npm run build` | Vite production build (`tsc -b` + `vite build`) | ~2s |
| `npx supabase test db` | pgTAP suite — RLS policies + `pg_jsonschema` CHECK constraint (11 assertions) | ~3s test runtime, but requires `supabase start` first which is **slow** (Docker image pulls on cold cache, ~1–3 min) |

The first five are fast and self-contained — they run against jsdom + MSW, no Docker, no real Supabase. The sixth (pgTAP) needs the local Supabase Docker stack up.

**Recommended split** (worth confirming with the user but a sensible default):

- **Job 1 — frontend** (fast, ~1 min total): the first five gates. Should give PR feedback in well under 2 minutes.
- **Job 2 — db** (slower, ~3–5 min total with cold Docker): just `supabase test db`.

These can run in parallel. Failing either fails the workflow.

## Things that are NOT secrets

CI doesn't need any secrets:

- The supabase publishable key is per-environment (local has its own; prod is only needed for deploy, which Vercel handles separately).
- The MSW + factory test layer means the frontend job never makes real HTTP requests.
- pgTAP runs against the *local* `supabase start` stack, which uses default-everything (no real OAuth, no real keys).

If you later add a job that pushes migrations to a real Supabase project (e.g., a "verify migrations apply cleanly to a Supabase preview project"), THAT would need secrets. Out of scope for v1 CI.

## Hard constraints (don't disrupt)

1. **Don't break the local commands.** All six gates should keep working as they do today. The CI workflow runs the same `npm` / `npx` commands; don't introduce CI-specific scripts that diverge from local.
2. **Node 20+.** The runbook mentions Node 20; the codebase uses native `crypto.randomUUID` (Node 19+) and `node:` import prefixes. Pin the matrix to Node 20 or 22.
3. **Don't add a new lint rule or test framework as part of CI setup.** That's scope creep. Stick to wiring what exists.
4. **The Vite plugin in `scripts/vite-supabase-env.ts` runs `npx supabase status -o env` at vite-config time.** It silently no-ops if the CLI errors. CI's `npm run build` will trigger this — make sure the plugin's `command !== "serve"` guard correctly skips during build (it does today). If you change the plugin, run a smoke `npm run build` locally to verify.

## Open questions / decisions to make

1. **Trigger:** PRs only, or PRs + pushes to main? Default suggestion: both — protects main from direct pushes too.
2. **Caching strategy:**
   - npm: `actions/setup-node`'s `cache: "npm"` is a one-line win.
   - Docker images for Supabase: GitHub Actions has limited Docker layer caching. The `supabase/setup-cli` action helps; first run pulls Postgres/GoTrue/Kong/etc. (~600MB). A persistent cache for these is possible but adds complexity. **Recommendation:** accept the cold-start cost on the db job for v1; revisit if it becomes painful.
3. **Concurrency:** cancel-in-progress on the same PR/branch? Yes — saves runner minutes when a PR pushes a new commit.
4. **Status check requirements:** post-CI, do we add branch protection on `main` requiring both jobs to pass? Recommend yes, but that's a one-click setting in repo settings; not part of the workflow file itself.
5. **Coverage:** explicitly out of scope unless the user asks. v1 doesn't have a coverage tool configured.

## Suggested workflow shape

Two suggested paths — pick whichever the user prefers:

### Option A — single workflow file, two jobs (recommended)

`.github/workflows/ci.yml`:

- **frontend** job:
  - actions/checkout
  - actions/setup-node@v4 (Node 20, cache: npm)
  - `npm ci`
  - `npm run lint`
  - `npx tsc -b --noEmit`
  - `npm test`
  - `npm run check:schema`
  - `npm run build`
- **db** job (parallel):
  - actions/checkout
  - actions/setup-node@v4 (Node 20, cache: npm)
  - `npm ci`
  - `supabase/setup-cli@v1` (or fall back to `npm ci` already including the CLI as a dev dep)
  - `npx supabase start` (long; consider `--workdir`)
  - `npx supabase test db`
  - `npx supabase stop` (cleanup, not strictly necessary)

### Option B — two separate workflow files

- `.github/workflows/frontend.yml` — runs always.
- `.github/workflows/db.yml` — runs on changes to `supabase/**` only (path-filtered).

**Tradeoff:** A is simpler and gives a single failing-or-not signal. B is slightly faster on PRs that don't touch DB stuff, at the cost of a more complex required-checks setup.

Default recommendation: **Option A**. Optimize later if path-filtering becomes worthwhile.

## Specific things to verify before declaring CI green

1. **`npm ci` on a clean cache** — confirm `package-lock.json` is committed and `npm ci` succeeds without writing to it.
2. **The `supabase` dev dep is sufficient** — `npx supabase` in the runner should resolve to the project-local install (`devDependencies.supabase`), so no separate setup step is required for the CLI binary itself. Only the Docker images are downloaded on demand by `supabase start`.
3. **`vi.stubEnv` in `src/test/setup.ts`** — the test setup stubs `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` before any test runs, so the supabase client doesn't throw under Vitest. Confirm this works in CI (no `.env.local` is present in the runner).
4. **`npm run build`** — the Vite plugin tries to call `npx supabase status` at config time. In CI it'll fail (no `supabase start`), but the plugin catches that silently and `npm run build` skips it via `command !== "serve"`. Verify the build job exits 0 even when Supabase isn't running.
5. **`supabase test db` cold cache** — the first run on a fresh runner pulls images. Don't be surprised by 2–3 minute startup. After Docker is up, the actual tests run in <1s.
